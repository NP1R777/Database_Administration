### 1. Оптимизация конфигурации PostgreSQL.
Для оптимизации работы СУБД, было принято решение подробнее познакомиться с главным конфигурационным файлом, а именно `postgresql.conf`. Вот основные настройки, которые влияют на производительность СУБД:
  - `shared_buffers` - параметр, отвечающий за объём оперативной памяти, выделяемый под буферный кэш PostgreSQL. Здесь сервер хранит часто используемые данные (таблицы, индексы) в оперативной памяти, чтобы минимизировать дисковые чтения. В нашем случае этот параметр имеет значение равное 128-ми мегабайтам. Так как объём оперативной памяти виртуальной машины равен 2-ум гигабайтам, то значение можно увеличить до 256-ти или 512-ти мегабайт, но делать этого не нужно, т.к. в предыдущей лаборатороной работе было выявлено, что этот параметр для СУБД в данной ситуации омтимален;
  - `work_mem` - парамет, задающий объём памяти для операций сортировки, хеш-таблиц и других операций внутри одного запроса. Для нашего случая, т.е. 2-ух гигабайт оперативной памяти, оптимальным значением будет 4-8 мегабайт;
  - `maintenance_work_mem` - параметр, определяющий объём памяти для операций обслуживания (VACUUM, CREATE INDEX, REINDEX, добавление внешних ключей). Эти операции выполняются реже, но требуют больше памяти. Для виртуальной машины с объёмом оперативной памяти 2 гигабайта, оптимальным значением будет 64-128 мегабайт;
  - `effective_cache_size` - параметр, который не выделяет память, а даёт планировщику оценку размера файлового кэша операционной системы. Этот параметр влияет на выбор плана запроса (например, планировщик предпочтёт индекс, если считает, что данные в кэше). Для машины с 2-мя гигабайтами оперативной памяти оптимальным размером будем чуть больше 1-го гигабайта.

После внесённых изменений в конфигурационный файл `postgresql.conf`, СУБД была перезагружена командой `sudo systemctl restart postgresql`. Далее было принято решение, через утилиту `psql` проверить, видит ли СУБД изменённые параметры. Посмотрим на скриншот ниже:

### 5. Автоматическая очистка и статистика (VACUUM, ANALYZE).
В `PostgreSQL` для поддержания производительности и актуальности статистики используются два важных механизма: `VACUUM` - очистка «мёртвых» строк и `ANALYZE` - сбор статистики. Их можно запускать вручную или доверить автоматическому демону `autovacuum`. Параметры демона `autovacuum` задаются в основном конфигурационном фале `postgresql.conf`. Вот его основные параметры:
  - `autovacuum` - параметр, указывающий на то, включён демон или нет;
  - `autovacuum_naptime` - параметр, определяющий частоту начала работы демона (как часто он начинает свою работу);
  - `autovacuum_vacuum_threshold` - параметр, определюящий минимальное количество строк, для запуска `VACUUM` по таблице;
  - `autovacuum_vacuum_scale_factor` - параметр, определяющий долю от размера таблицы, которая будет добавлена к порогу;
  - `autovacuum_vacuum_cost_delay` - параметр, определяющий задрежку между операциями чистки. Применяется для того, чтобы избежать перегруза системы.

  - `stats_reset` - время последнего сброса всей статистики.

Осталось понять два момента, один из которых: как работает autovacuum? Автовакуум запускается, когда для таблицы выполняется условие:
```
dead_tuples > autovacuum_vacuum_threshold + autovacuum_vacuum_scale_factor * n_live_tup
```
По умолчанию: `50 + 0.2 * число_строк`. Если в таблице 1000 строк, порог = 250 мёртвых строк. Когда количество мёртвых строк превышает порог, автовакуум обрабатывает эту таблицу. Параметр `autovacuum_vacuum_cost_delay` ограничивает скорость очистки, чтобы не мешать основной работе. В высоконагруженных системах его уменьшают или отключают, но тогда `VACUUM` может стать тяжёлым.
Второй момент: что такое мёртвые строки? Рассмотрим пример:
```
UPDATE test SET name = 'Bob' WHERE id = 1;
```
Что происходит внутри `PostgreSQL`:
  - `PostgreSQL` создаёт новую версию строки с name = 'Bob';
  - Старая версия помечается как мёртвая (dead) для всех транзакций, кроме тех, которые начались до обновления;
  - Но старая версия физически остаётся в файле таблицы.

Так вот эта старая версия такой строки и называется мёртвой.
