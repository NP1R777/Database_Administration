## Ход работы
### 1. Утилиты резервного копирования.
В данном шаге необходимо рассмотреть отличия между утилитами резервного копирования `pg_dump` и `pg_basebackup`.
  - `pg_dump` - создаёт логическую копию одной базы данных (или части её) в виде SQL-скрипта или архивного формата.
    Она выгружает структуру и данные, преобразуя их в команды CREATE, INSERT и т.д. По сути, с помощью этой команды
    просто создаётся большой SQL-скрипт, который при запуске создаёт точную копию базы данных.
  - `pg_basebackup` - создаёт физическую копию всего кластера PostgreSQL (всех баз данных) на уровне файлов данных.
    При использовании этой утилиты создаётся не SQL-скрипт, а файлы баз данных, а точнее кластера баз данных.

Говоря про сценарии использования каждой утилиты можно сказать, что `pg_dump` используется в случаях, когда надо перенести одну базу данных или даже её часть. Как правило это происходит в небольших проектах. `pg_basebackup` используется для резервного копирования всего кластера с базами и нужна, если планируется масштабный перенос данных (например с одного сервера на другой)

### 2. Создание резервной копии.
Для создания дампа базы данных была выбрана база данных `test_db` из лаборатороной работы №1 и утилита `pg_dump`. Для описания всего процесса, ниже представлен
скриншот:

![Создание дампа базы данных](https://github.com/NP1R777/Database_Administration/blob/main/Laba2/images/1.png)

Как вы можете видеть, сначала создать дамп не удалось, т.к. у пользователя `postgres` не было прав на запись в каталог `/var/backups/`. В связи с этим, было принято решение предоставить ему права с помощью команды `chown`, и после этого, дамп был создан и помещён в соответствующий каталог. Теперь более подробно пройдёмся по процессу создания дампа базы данных:
  - Параметр `-U` - здесь мы явно указываем пользователя, от которого делается дамп, чтобы иметь доступ ко всем таблицам;
  - Параметр `-d test_db` - явно указываем дамп какой базы данных хотим сделать;
  - Параметр `-Fc` - обозначает формат дампа. В данном случае он кастомный. О других параметрах более подробно поговорим ниже;
  - Параметр `-f` - задаёт итоговое название файла. В данном случае был использован для того, чтобы задать метку времени.

Теперь более подробно поговорим о видах дампов баз даннных:
  - `-Fc` - кастомный дамп. Позволяет создать дамп меньшего размера т.к. на выходе мы получаем бинарный файл, который поддерживает выборочное и параллельное восстановление;
  - `-Fp` - создаёт обыячный файл из SQL-команд;
  - `-Fd` - создаёт каталог, содержащий несколько файлов: toc.dat (оглавление), файл с метаданными, и отдельные файлы для каждой таблицы. При создании можно указать уровень сжатия;
  - `-Ft` - Обычный tar-архив (несжатый), внутри которого лежат: toc.dat (оглавление), restore.sql (метаданные), и по одному файлу на каждую таблицу с данными.
