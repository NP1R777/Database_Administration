# Лабораторная работа №1: "Базовая настройка PostgreSQL на Debian". Отчёт.
## План работы:
Настроить окружение, установить PostgreSQL в Debian, освоить базовые приёмы 
администрирования системы и СУБД.

## Ход работы
### 1. Подготовка среды.
Для выполнения лабораторной работы был скачан образ Debian, операционной системы на базе Linux. После чего была проведена установка данной системы в приложении
VirtuaBox, для удобного взаимодействия с системой. После установки системы было принято решение проверить, всё ли в порядке с её обновлениями. Для проведения
такой проверки, в терминале Debian были введены следующие команды:
  1. "apt-get update" - команда, проверяющяя, есть ли обновления для системы;
  2. "apt-get upgrade" - команда, устанавливающая в систему обновления, если они имеются.

поиска строк по заданному шаблону, в вышеописанном файле была найдена строка, говорящая о том, что запись `postgres` в ходе установки была успешно создана:
![Содержимое файла passwd]()

Как видно из скриншота выше, запись в файле `passwd` существует, а это значит, что учётная запись `postgres` в ходе установки СУБД была успешно создана. Данная

### 4. Первичная настройка конфигурационных файлов.
После разбора с учётными записями была поставлена следующая задача - ознакомиться с базовыми конфигурационными файлами PostgreSQL. Давайте по порядку:
  1. `pg_hba.conf` — это основной конфигурационный файл PostgreSQL, который управляет аутентификацией клиентов. `HBA` расшифровывается как
      «host-based authentication» (аутентификация на основе хоста). Простыми словами, это «список допуска», который проверяет каждое входящее подключение
      к серверу. В данном файле были изменены следующие настройки:
      ![Настройки в файле pg_hba.conf]()

      Если говорить подробно, то вот что значат данные поля в файле:
        1. `TYPE` - тип соединения;
        2. `DATABASE` - имя целевой базы данных;
        3. `USER` - имя пользователя (роли) БД;
        4. `ADDRESS` - IP-адрес клиента или диапазон;
        5. `METHOD` - метод аутентификации.
     Для того, чтобы ничего не сломать и не повредить работе сервера копанием в конфигурационных файлах, было принято решение изменить только метод
     аутентификации на `trust` в одной из записей, чтобы можно было входить в систему без пароля.

  2. `postgresql.conf` - основной файл конфигурации сервера СУБД PostgreSQL, в котором определяются параметры, влияющие на работу сервера
     (память, производительность, журналирование и др.). В данном файле были изменены следующие настройки:
     ![Настройки в файле postgresql.conf]()

     Если подытожить, был изменён порт со стандартного `5432` на `5433`, так же был изменён параметр, отвечающий за максимальное количетсво разрешённых
     подключений. Его значение было изменнено со 100 на 50.

После вышеописанных манипуляция сервер был перезагружен с помощью команды `systemctl reload postgresql` и проверен его статус командой
`systemctl status postgresql`. Результат можно увидеть ниже:
![Проверка работы сервера после изменений в файлах конфигурации]()

### 5. Управление сервисом.
В данном пункте лабораторной работы необходимо было познакомиться с системной утилитой Linux `systemctl`. Данная утилита позволяет контролировать системные
процессы, и вот базовые команды, которыми пригодились в ходе выполнения данной лаборатороной работы:
  1. `ststus` - проверка состояния того или иного системного процесса;
  2. `reload` - перезагрузка процесса;
  3. `restart` - полный перезапуск процесса;
  4. `start/stop` - начало/остановка процесса.
В предыдущем шаге было показано как с помощью данной утилиты можно увидеть состояние сервера на текущий момент. Автозапуск сервера `postgresql` был включён
автоматически.

### 7. Знакомство со схемами.
В PostgreSQL схема — это пространство имён (контейнер) внутри одной базы данных, которое группирует объекты: таблицы, представления, функции, типы и т.д.
Схемы решают несколько задач:
  1. Организация — логическое разделение объектов по функциональности;
  2. Управление правами — можно дать доступ ко всем объектам схемы одной командой;
  3. Изоляция — в одной базе данных могут существовать объекты с одинаковыми именами, если они лежат в разных схемах
     (например, public.users и admin.users — две разные таблицы).
Если говорить про разницу схемы и базы данных, то можно сказать так: база данных, это изолированная среда, в которой есть свои таблицы, связи, имена и т.д. Схема
же - это логическая группировка внутри одной БД.
Для дальнейшего выполнения лаборатороной работы, необходимо было создать новую схему и настроить у пользователя права на использование данной схемы:

### 9. Настройка локальных и сетевых подключений.
Для того, чтобы настроить доступ для любых подключений на сервер, необходимо зайти в уже известный файл `postgresql.conf`, раскомментировать параметр
`listen_addresses`, и вместо значения `localhost` поставить `*`. Это означает, что теперь к серверу БД могут подключаться все. Теперь подробнее разберём формат
строки в файле `pg_hba.conf`:
  1. `host` — означает TCP/IP-соединение (не локальный сокет);
  2. `all` — правило действует на все базы данных. Можно указать конкретную, например, test_db;
  3. `all` — правило действует на всех пользователей. Можно указать limited_user.
  4. `192.168.1.100/32` — адрес. Это самая важная часть для безопасности. `/32` — это конкретный один IP-адрес.
     Можно указать целую подсеть, например `192.168.1.0/24`.
  5. `scram-sha-256` — метод шифрования пароля. Это самый безопасный из распространённых.
На скриншотах ниже продемонстирован результат подключения сервера PostgreSQL, развёрнутого на вирутальной машине к локальной машине через `pgAdmin 4`:

### 10. Журналирование (logging).
Для корректного хранения логов, которые пишутся в процессе работы СУБД, необходимо было поменять настройки логирования в файле `postgresql.conf`. Для удобного
логирования, в файл были внесены следующие изменения:
  1. `logging_collector = on` - включен сбор данных в файл;
  2. `log_directory = 'log'` - указана конкретная директория для хранения логов, которая находится по пути: `/var/log/postgresql/`;
  3. `log_min_duration_statement = 100` - включено голирования запросов, выполняющихся дольше 100 миллисекунд.
После внесения данных изменений, сервер был перезапущен. На скриншоте ниже можно увидеть отображающиеся логи:
